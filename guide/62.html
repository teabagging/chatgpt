Develop an algorithm that can detect the unique elements in two unsorted arrays of integers and output their count. The algorithm should also handle cases where arrays are of different lengths. Further, implement this without using any preexisting library function for handling collections or arrays. [11, 17, 8, 29, 32, 45], [2, 8, 11, 42]

Let's break down the thought process to arrive at the algorithm and its implementation.**1. Understanding the Problem:**The core goal is to find the *unique* elements present across *two* unsorted integer arrays. "Unique" here means elements that appear only once *in the combined set of elements from both arrays*. The output should be the *count* of these unique elements. We also need to handle arrays of different lengths and avoid using library functions for collections (like sets, dictionaries, or built-in array methods that simplify uniqueness checks).**2. Initial Brainstorming (Brute Force/Naive Approaches):*** **Nested Loops (Compare every element with every other element):** The most basic approach that comes to mind for uniqueness is to compare each element with every *other* element. In the context of *two* arrays, we could try combining them conceptually and then checking for uniqueness within that combined set. This will likely involve nested loops and comparisons.* **Sorting (Maybe?):** Sorting could help in identifying duplicates easily. If we sorted both arrays and then combined them, duplicates would be adjacent. However, sorting itself might be considered a library-like function depending on how strictly the "no libraries" rule is interpreted (though often sorting algorithms are considered fundamental). Also, sorting is likely more computationally expensive than necessary if we *only* need to count unique elements. Let's keep this in mind as a potential optimization, but not the primary focus initially.* **Frequency Counting (Hash Map/Dictionary Idea - but we can't use pre-built ones):** The most efficient way to count unique elements typically involves using a hash map (or dictionary) to store the frequency of each element. If the count is 1 after processing both arrays, the element is unique. Since we can't directly use a hash map, we need to simulate its behavior. This is a very promising direction.**3. Refining the Frequency Counting Approach (Without Hash Maps):**Since we can't use hash maps directly, how can we simulate frequency counting? We need a way to:* **Store elements we've encountered.*** **Keep track of how many times we've seen each element.**A simple way to do this without a hash map is to use a *list* (or an array, if we were pre-allocating, which isn't strictly necessary here in Python, but good to consider in other languages). We can iterate through the combined elements of both arrays. For each element, we can check if we've already seen it in our "seen" list.**4.
